[DESCRIPTION]
Load when creating, reviewing, or modifying a skill. Use for designing procedural skill content, authoring data sections, separating behavior from data, converting user corrections or failed workflows into better skill rules. All skills generated by this manager must use compact-language format.

[BEHAVIOR]
skill := folder containing skill.md that improves future agent behavior by defining when it applies, what workflow to follow, what to try first, what to avoid, known pitfalls

skill_format:
  - skill = folder `<name>/skill.md`
  - ! never flat .md directly under skills directory → invalid, not discovered
  - structural blocks:
    - `[DESCRIPTION]` → required, must appear first
    - `[BEHAVIOR]` → optional, procedural guidance
    - `[DATA]` → optional, persistent facts
  - prereq: at least one of `[BEHAVIOR]` `∨` `[DATA]` must be present

description_rules:
  - short, trigger-based
  - must state:
    - `Load when` → user intent, topic, domain, or task that activates the skill
    - `Use for` → what the skill helps accomplish
  - concrete trigger words user likely to say
  - names relevant tools, systems, domains
  - concise → no long instructions
  - ! no inventories, credentials, logs, changelogs
  - ! never compress description → natural language, trigger words must match user phrasing
  ex:
    Load when the user wants Node-RED, MQTT, Zigbee, or smart-home actions. Use for MQTT workflows, Zigbee2MQTT troubleshooting, SSH access patterns, and safe smart-home command guidance.

behavior_rules:
  - procedural guidance → workflow, decision rules, strategy, pitfalls, fallbacks, validation, safety constraints
  - answers:
    - first: what to do
    - ! what not to try first
    - what assumptions are dangerous
    - what signals indicate success `∨` failure
    - stop: when to stop and ask user

behavior_vs_data:
  - `[BEHAVIOR]` → how to work
  - `[DATA]` → what is true
  - skill may contain only `[BEHAVIOR]` `∨` only `[DATA]` `∨` both
  - skill needs persistent domain facts → define in its own `[DATA]` section
  - ! never create separate skill just for data

recommended_structure:
  - order for most operational skills:
    1. purpose
    2. preferred workflow
    3. first checks `∨` first strategy
    4. known pitfalls
    5. fallbacks
    6. validation
    7. stop conditions
    8. examples
  - data-only skills → concise `[DATA]` with `key = value` facts
  - ! avoid forcing structure on very small skills → concise procedural rules > many headings

skill_storage:
  - two scopes:
    - @global → shared across all projects
      path: `{GLOBAL_SKILLS_DIR}/<name>/skill.md`
    - @project → scoped to current project
      path: `{PROJECT_SKILLS_DIR}/<name>/skill.md`
  - both use same layout: `<name>/skill.md`
  - ! never flat .md file directly under skills directory

create_or_edit:
  prereq: active skill `[BEHAVIOR]` `∧` `[DATA]` already in context
  - ! do not read file from disk unless modifying
  - modifying skill → read file first → write updated version
  - commands:
    - `mkdir -p {GLOBAL_SKILLS_DIR}/<name>`
    - `mkdir -p {PROJECT_SKILLS_DIR}/<name>`
  - variables resolve to real paths automatically
  - `{SKILL_DIR}` → reference skill's own folder inside content (e.g. bundled scripts)

injectable_variables:
  - `{APP_HOME}` → BlazeAI app home directory
  - `{WORK_DIR}` → current working directory
  - `{OS_INFO}` → human-readable OS description
  - `{SKILL_DIR}` → skill's own folder on disk
  - `{GLOBAL_SKILLS_DIR}` → global skills root directory
  - `{PROJECT_SKILLS_DIR}` → current project skills root directory

escaping:
  - `{VAR}` → injectable, resolves at build
  - `\{VAR\}` → literal, backslash stripped at build
  - `\[BLOCK\]` → literal block
  - `\\` → literal backslash
  - reserved chars needing escape in text: `[` `]` `{` `}` `→` `!` `#` `@`

scope_selection:
  - @global when:
    - skill generic `∧` reusable across projects
    - about personal tools / preferences
    - cross-project knowledge (network, backup scripts, music player, smart home)
  - @project when:
    - skill specific to this codebase
    - build rules, architecture, deploy, project conventions, project-specific paths `∨` names
  - unsure → ask user

generalize_to_global:
  - promoting project skill to global:
    - remove absolute paths → use `{SKILL_DIR}` `{APP_HOME}` `{GLOBAL_SKILLS_DIR}` `{PROJECT_SKILLS_DIR}` `∨` generic descriptions
    - remove project / branch names `∧` project-specific conventions
    - write examples generically → apply to any project
    - nothing useful remains after generalizing → keep @project

restore_builtins:
  - builtins (skill-manager, customize_me, setup_helpers) → seeded into `{GLOBAL_SKILLS_DIR}` on startup
  - restore builtin to factory version → delete folder → restart BlazeAI

compact_language:
  compact_language := operator-based format for compressing natural language into structured, token-efficient instructions
  - all skills generated by this manager must use compact-language
  - ! never generate skills in natural-language prose
  - ! never compress `[DESCRIPTION]` → natural language, trigger words must match user phrasing

  principles:
    - no prose → each info type has one format
    - no narrative connectors → order + operators carry relations
    - only notation from math, logic, code → no invented symbols
    - one operator = one meaning
    - consistency > extreme compression

  hierarchy_and_layout:
    - indent 2 spaces → hierarchy
    - `- item` → unordered list, one per line
    - `1. 2. 3.` → ordered list
    - one line = one info unit

  operators:
    atribuire_definitie:
      - `=` — fact / value
        ex: `key = value`
      - `:=` — nominal definition
        ex: `name := value`

    implicatie_trigger:
      - `→` — if X then Y (most used operator)
        ex: `condition → action`
      - compound conditions:
        - `cond1 ∧ cond2 → action` (and)
        - `cond1 ∨ cond2 → action` (or)

    conditional_ramuri:
      - `cond ? a : b` — ternary
      - multiple branches → separate lines with `→`, last with `else →`

    negare_prohibitie:
      - `!` prefix — do not / avoid
        ex: `!action`
      - absolute prohibition → `! never action`
      - intensity via words: `never` `avoid` `rarely`

    apartenenta_clasificare:
      - `∈` — X is member of Y
        ex: `X ∈ Y`
      - `∉` — X is not member of Y
        ex: `X ∉ Y`
      - `⊆` — A is subcategory of B
        ex: `A ⊆ B`

    context_scope_tag:
      - `@` prefix — labels context, scope, or category
        ex: `@global` `@project` `@scope: global`
      - `@` labels who entity is `∈` asserts membership in a set

    exemplu:
      - `ex:` prefix — example
      - placed below rule it illustrates, at one deeper indent
      - may use any operator

    comentariu_nota:
      - `#` prefix — explanatory note, rare
      - if worth keeping permanently → becomes rule or fact

  keywords_explicite:
    - natural language prefixes for info types without native symbol operator
    - `stop: condition` — stop when
    - `valid: condition` — success confirmed when
    - `fallback: action` — try if rest failed
    - `first: action` — try first
    - `prereq: condition` — required before continuing

  info_type_format:
    - trigger / activation → `condition → action`
    - rule / constraint → `condition → action` `@` `!action`
    - procedure → `1. 2. 3.` ordered steps, branches via `cond ? a : b` `@` lines with `cond →`
    - fact / data → `key = value`
    - definition → `name := value`
    - example → `ex:` below illustrated rule
    - warning / pitfall → `! action` + `never` / `avoid`
    - scope / context → `@...` prefix
    - fallback → `fallback: action` `@` `else → action`
    - stop condition → `stop: condition`
    - validation → `valid: condition`
    - prerequisite → `prereq: condition`

  ! never compress:
    - trigger words in descriptions → natural language, must match user phrasing
    - examples → natural language allowed, prefix `ex:`
    - notes (`#`) → natural language allowed, rare
    - ambiguous rule → rephrase mixed: compact structure + short clarification after `#`

  conventions:
    - names / scopes → lowercase, `_` for spaces
    - actions → imperative: `read file` `ask user` `create folder`
    - conditions → declarative: `file exists` `user confirms` `skill missing`
    - ! no diacritics in compact content
