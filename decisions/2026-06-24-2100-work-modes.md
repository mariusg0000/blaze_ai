# Session Decision Summary: Work Modes Feature

Date: 2026-06-24 21:00
Base commit: ba3c825

## Context
The user requested a persistent "work modes" feature for BlazeAI. Each mode binds a model, an optional directive, and a name. The user cycles modes with Tab at the prompt. The directive is injected into the last message sent to the LLM (volatile, never persisted in session.json). Modes are defined in config.json.

## Changes Made
- Added `Mode` struct and `Modes []Mode` + `LastMode string` fields to config.
- Added mode validation: unique names, valid model format/provider, `last_mode` references existing mode.
- Pre-created `"default"` mode at first-run setup.
- Added `CurrentMode *config.Mode` to Agent struct, initialized from `LastMode`.
- Added `SetMode(name)`, `NextMode()` methods for mode switching.
- Extended `SetModel` to update `CurrentMode.Model` when a mode is active.
- Added `injectDirective()` function that appends `[MODE DIRECTIVE]` to a copy of the last message.
- Modified `RunTurn` to inject directive between `StripReasoningFromPayload` and `Provider.Stream` (after debug prompt.json write).
- Added raw-mode Tab detection in `reader.go` via `golang.org/x/term`.
- Split `Run()` into `runTTY()` (direct ReadEvent, no goroutine) and `runNonTTY()` (existing goroutine).
- Prompt label shows `[USER/<model>|<mode>]` when mode active.
- Updated `customize_me.md` skill with Work Modes section.

## Decisions And Rationale
- **Directive injection point**: Between StripReasoning and Stream, after prompt.json write. This ensures session.json never contains the directive and prompt.json (debug) matches the persisted session state.
- **TTY vs non-TTY split**: TTY uses direct `ReadEvent()` calls (no goroutine) to avoid raw-mode during streaming. Non-TTY keeps the existing goroutine approach for "queue next input" feature.
- **`last_mode` validation**: Hard error (no fallbacks) per spec mandate. An invalid `last_mode` stops the runtime with a clear message.
- **Mode model priority**: When a mode is active, its model takes priority over `LastModel` in NewAgent initialization.
- **SetModel extends to mode**: When a mode is active and user runs `/model`, the mode's model is updated so the change persists with the mode.

## Implementation Approach
1. Config: Mode type + validation functions + error vars.
2. First-run: pre-create default mode after role assignment.
3. Runtime: CurrentMode field, SetMode/NextMode methods, injectDirective function, RunTurn modification.
4. Console: ReadEvent in reader.go (raw mode on TTY), runTTY/runNonTTY split, prompt label update.
5. Skill: Work Modes section teaching LLM to create/edit modes.

## Alternatives Considered
- **Raw mode with goroutine**: Rejected because it causes raw mode during streaming, breaking Ctrl-C SIGINT. TTY mode uses direct ReadEvent calls instead.
- **liner/readline library**: Rejected by user as not worth the dependency weight.
- **`last_mode` auto-reset**: Rejected in favor of hard error per "no fallbacks" spec mandate.
